# Authoring hints

This README is gitignored — it's local scaffold material, regenerated by `/forge init`. Don't put project-specific knowledge here; that belongs in the per-consumer hint files alongside this README.

## All hint files are optional

Forge drives correctly against a bare scaffold — every default is sane for an unauthenticated public site. Hints are purely additive: write them only when you have project-specific knowledge the agents can't discover on their own.

A reasonable approach is to drive a task with no hints, see what the agents struggled with, then author the hint that would have helped. Iterate.

## File-per-consumer convention

Each hint file is named after the agent or skill that reads it. Forge looks for files matching these specific names:

- `forge.md`          — read by the `/forge` skill
- `driver.md`         — read by `forge:driver`
- `snippet-author.md` — read by `forge:snippet-author`
- `spec-writer.md`    — read by `forge:spec-writer`
- `spec-verifier.md`  — read by `forge:spec-verifier`

## What each file typically covers

Hints are written in your own words. There are no required headings; forge reads them as natural-language instructions. The headings below are common-but-optional patterns — use them as a starting point and adapt.

### `forge.md` — the project-level overview

Worth writing first if your site has auth or any setup/teardown forge can't figure out on its own.

- **Authentication** — what env vars forge needs to log in (e.g. `PORTAL_USERNAME`, `PORTAL_PASSWORD`, `TENANT_ID`). Just declare the keys; the actual values live in local `.env` files which are gitignored, not in hints.
- **Test accounts available** — which logins forge can use, and the differences between them ("admin can see everything; member is read-only"). For apps where new test users can be minted on-demand (e.g. SQL insert), describe how.
- **Adding another test account to the rotation** — what to do when forge needs more concurrent logins than are currently available. Often "wait, the set is fixed"; sometimes a recipe.
- **Setup before each run** *(optional)* — anything that should happen before a run starts beyond the default. Write it in your own words: "create a fresh test user with this SQL," "wipe the events table," "don't reset any state — runs share state intentionally." Forge reads this as instructions to itself. By default it wipes cookies + localStorage + sessionStorage from the browser profile (covers most leakage); opt out explicitly, extend by adding your own steps.
- **Teardown after each run** *(optional)* — anything that should happen at the end of a run that forge can't see (logout endpoints, account cleanup, third-party integrations to reset). There's no default teardown.

### `driver.md` — what the driving agent needs to know about your app

The biggest quality-of-output lever, especially for complex UIs.

- Common routes / URL patterns the driver will navigate.
- App structure ("this is a planner with these tabs / this is an e-commerce with these flows").
- Selector conventions ("we use `data-test` attributes throughout — prefer them") or known-stable selectors for the most-used controls.
- Known gotchas — UI quirks, timing issues, things that look broken but aren't, anything the driver would otherwise have to discover by trial and error.

### `snippet-author.md`, `spec-writer.md`, `spec-verifier.md`

Usually quite small. Project-specific exceptions to the universal defaults that ship with the agents. For example:

- "Our checkout flow can take 30s on slow CI; allow longer waits."
- "Specs in this project name themselves `<feature>-<scenario>.spec.ts`."

If you don't have any deviations, don't author the file. Defaults apply.

## Tips

- **Write in plain language about your project.** "We use `data-test` selectors" or "checkout takes 30s on slow CI" — describe what's true of the app, in your own words.
- **Keep hints declarative.** Describe the pattern: "use direnv to load env". The agents apply the pattern themselves; you don't need to enumerate the steps.
- **Credentials go in `.env` (gitignored), not hints (committed).** Hints declare which env keys exist; the values live in local `.env` files alongside.
- **Hints encode project-specific knowledge only.** Universal best practices like "snippets are idempotent" and "specs run from a cold start" are already enforced by default — your hints are for the things only your project knows.
