# Authoring hints

This README is gitignored — it's local scaffold material, regenerated by `/forge init`. Project-specific knowledge belongs in the hint files alongside this README (`forge.md`, `curator.md`), not in this README itself.

## The hint files are optional

Forge drives correctly against a bare scaffold — every default is sane for an unauthenticated public site. `/forge init` creates the two hint files **empty**; fill them in only when you have project-specific knowledge the agents can't discover on their own.

A reasonable approach: drive a task with no hints, observe what the agents missed or hardcoded, write the hint that would have helped. Iterate.

## Two files, one boundary — operate vs curate

Forge loads exactly two hint files, by name:

- **`forge.md`** — read by the **lead and the driver**. Everything about *operating the app*: env contract, accounts, setup/teardown, app structure, routes, selectors, gotchas, spec deviations. The high-leverage file.
- **`curator.md`** — read by the **curator**. Snippet-authoring conventions only. Usually stays empty.

Why the boundary is operate-vs-curate (and not one file per agent): the driver and lead share the app's world — the lead is the driver's investigative backstop and routes its check-ins using the same selectors and gotchas. The curator works on a different plane — it never touches the browser; it turns the driver's trace into library code — so its conventions live apart.

**Cross-access via pointer.** To opt an agent into a file it doesn't load by default — say you keep a large selector inventory in its own file — just say so *inside* the hint: "the selector inventory lives in `selectors.md`." Agents follow in-hint pointers. The default is choosy; you override per project.

## Where the leverage is

The most consequential content is **`forge.md`'s selector inventory**. Here's the mechanism:

When the curator sees the selector the driver used (from the trace) for a named element class, it tends to author a snippet whose scope maps 1:1 to that class ("click first product card"). Narrow snippets compose well; the driver, composing the spec, parameterises over snippet args rather than embedding the specific value it happened to encounter.

Concretely: in field tests, the same drive task produced two very different specs depending on hints:

- **No selector hint:** spec embeds `https://shop.example.com/product/<specific-uuid>`. Works once. Breaks when inventory rotates.
- **`forge.md` lists `a[data-test^="product-"]` for product cards:** spec composes `search-for-product → open-first-search-result → add-product-to-cart`. Works repeatedly. Survives inventory rotation.

The selectors didn't change *what the driver could do* — the bare driver passed both tasks. They changed *the shape of the resulting library*, which changed how the spec composed, which changed whether the spec was re-runnable.

So when investing time in hints, lead with the canonical selectors per element class in `forge.md`. Spec-mode compositionality and re-runnability follow from there.

## What each file typically covers

Hints are written in your own words. There are no required headings; forge reads them as natural-language instructions. The patterns below are common starting points — adapt as needed.

### `forge.md` — operating the app

Everything the lead and driver need to operate your project.

**App-driving knowledge (the driver's core need):**

- **Canonical selectors per element class.** "Search submit: `button[data-test="search-submit"]`", "Product card: `a[data-test^="product-"]`", "Cart icon: `a[data-test="nav-cart"]`". One selector per element class. The driver uses these to act; the curator uses them to scope each snippet to one concern.
- **Routes / URL patterns** the driver will navigate. "Home `/`, category `/category/<slug>`, product `/product/<slug>`, checkout `/checkout`." Lets the driver build URLs without inspecting the page first.
- **App shape** — a sentence or two: "Vue 3 SPA with paginated catalog, real form validation, multi-step checkout."
- **Known gotchas** — UI quirks the driver would otherwise discover by trial and error (and that the lead reaches for when routing a check-in). "Search box debounces 300ms — wait for network settle," "Cart badge updates async via the Pinia store — wait on the badge text, not URL change," "Password-strength meter blocks register submit until 'Strong'."
- **Spec-mode deviations** *(spec mode only)* — naming, imports, or verification/reset idioms. "Specs name themselves `<feature>-<scenario>.spec.ts`." Most projects need none; the spec defaults cover them.

**Run-level knowledge (the lead executes this; the driver uses the env/account parts):**

- **Authentication / env contract** — declare the env keys forge needs by name (e.g. `PST_EMAIL`, `PST_PASSWORD`, `TENANT_ID`). Don't include values; don't describe where files live. If a wrapping recipe is needed to load env vars before each command (each Bash invocation is its own shell), include it inline. Prefer a `direnv exec <profile> --` shape where the project allows — it's pre-allowed for forge's agents, so runs stay prompt-free. Other shapes (e.g. `set -a && source .env && set +a &&`) work too, at the cost of a one-time permission approval when the driver first uses them. The driver applies the recipe; you don't need to explain why. Map persona keywords to env-key names: "admin → ADMIN_USERNAME / ADMIN_PASSWORD; user → USER_USERNAME / USER_PASSWORD."
- **Test accounts available** — which logins forge can use and what differences matter. "admin sees everything; member is read-only." For apps where new test users can be minted on-demand, describe how.
- **Setup before each run** *(optional)* — write this only when the SUT has real server-side state worth resetting before each run. "Run `bun run db:seed`," "wipe the events table," "don't reset anything — runs share state intentionally." For read-mostly or public sites, skip.
- **Teardown after each run** *(optional)* — same shape, fires at end-of-work. There's no default teardown; opt-in only.

### `curator.md` — snippet conventions

Usually quite small or absent. Project-specific exceptions to the universal snippet-authoring defaults the curator ships with:

- "Snippets should treat `process.env.LOCALE` as required."
- "Compose multi-step flows from single-concern snippets rather than fusing them."

If you don't have deviations, leave it empty. Defaults apply.

## Tips

- **Write in plain language about your project.** Describe what's true of the app — "we use `data-test` selectors throughout," "checkout takes 30s on slow CI," "this is a Vue 3 SPA with a Pinia store."
- **Lead with selectors.** Selector vocabulary is where hints earn their keep, especially for spec-mode work and any flow you intend to re-run.
- **Credentials never live in hints (committed).** Hints declare which env key NAMES exist for which persona; values stay in your local env layer (direnv, `.env`, your secrets manager — your choice).
- **Universal best practices are already enforced by agent defaults.** Snippets are idempotent, specs run from cold start, the driver waits on state rather than URL changes — agents do these by default. Hints encode the things only your project knows.

## For AI-assisted hint authoring

If you're asking Claude (or another AI) to generate hints for a codebase, point it at this file and at the app itself. Lead with the selector inventory, declare the env contract, and skip `curator.md` unless you have specific snippet-convention deviations. A good `forge.md` reads like a senior engineer's notes after an hour of clicking around the app and reading its docs.

### Starter prompt

Paste this into Claude (or your AI of choice) at the start of a project session. Adjust the bracketed parts to fit your codebase; otherwise it's ready to use as-is.

> Read this project's source code and any available documentation. Based on what you find, draft `forge/hints/forge.md` for me — the single "operate the app" hint that forge's lead and driver read. Cover:
>
> - **App shape** — one or two sentences naming the frontend framework, UI library (MUI, Kendo, AntD, Bootstrap, etc.), and overall structure.
> - **Common routes** — the route patterns the driver will navigate (e.g. `/login`, `/events`, `/events/:id/...`). Include only what's actually present in the routing config.
> - **Canonical selectors per element class** — one selector per UI element class (search box, primary button, item card, etc.). Prefer `data-test*` / `data-testid` / stable IDs where they exist; fall back to role + name only when nothing more stable is present.
> - **Known gotchas** — anything the driver should be told upfront. Examples: component libraries that intercept pointer events (require `dispatchEvent('click')`), virtual lists with settle animations, dynamic IDs containing special characters, async stores that update after URL change, label/dropdown substring overlaps that need `{ exact: true }`.
> - **Env contract** — what env var NAMES forge expects to find in `process.env` for each persona (usernames, passwords, API keys, tenant IDs). Don't include values; just declare the key names. Map persona keywords to env key names: "admin → ADMIN_USERNAME / ADMIN_PASSWORD; user → USER_USERNAME / USER_PASSWORD."
> - **Authentication / test accounts** — what login the app expects, where test credentials are documented, any single-session-per-user constraints or rate limits.
> - **Setup / teardown before/after each run** *(optional)* — if the project needs SQL seeding, a database reset, or other state prep/cleanup the agent defaults won't cover, describe it in plain language.
>
> Use plain language, not configuration syntax. If something isn't documented and isn't obvious from the code, note the gap rather than guessing. Skip `curator.md` unless you spot a clear snippet-convention deviation worth recording — the curator defaults usually cover it.
